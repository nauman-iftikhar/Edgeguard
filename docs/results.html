Experiment Overview

We estimated π using the Monte Carlo method with MPI parallelization. Each rank independently generates random points and checks if they fall inside a unit circle. A single MPI_Reduce at the end combines results — making this an embarrassingly parallel workload with minimal communication overhead.

We tested 6 core counts (1, 2, 4, 8, 16, 32) across 5 problem sizes (1M to 10B points), running each configuration 3–5 times on an idle cluster with active fan cooling. Pi5 was excluded from compute — all 32 cores are from 8× Pi3 nodes (4 cores each).

Finding the Parallelization Limit (Amdahl's Law in Practice)

Prof. Baun's key challenge: "Find the point where the program cannot be parallelized anymore." We systematically investigated where our cluster hits this limit and what causes it. Single runs were used for preliminary investigation across problem sizes from 1B to 25B.

⚠️ Parallelization Limit Found: 8 Cores (2 Nodes)

At all problem sizes tested, 8 cores is the effective parallelization limit on our cluster. Beyond 8 cores, crossing from 2 to 4 physical nodes introduces MPI communication overhead that exceeds the compute gain — exactly the serial fraction Amdahl's Law identifies. Adding more cores (16) provides no meaningful benefit, while 32 cores recovers by utilizing all 8 nodes symmetrically.

🔍 Answer to Prof. Baun's Question

Problem size effect: Increasing from 1B to 25B (25× larger) improved 8→16 core performance by only 0.4% → 2.5% — a marginal effect. To achieve clear 2× improvement at 16 cores, problem sizes of ~500B+ would theoretically be needed, which is impractical on this hardware.

Switch topology effect: Rearranging switches improved latency by 42% and reduced jitter by 3.4×, with slight improvement in 8→16 performance (1.1% → 2.7% at 10B). However the plateau persisted.

Final conclusion: The plateau is primarily a fundamental MPI communication topology effect when crossing from 2 to 4 physical nodes. A single switch connecting all 9 devices is the recommended next step to fully eliminate this bottleneck.

Key Findings

1. Near-ideal scaling for 1→2→4→8 cores

At 1B and 10B points, speedup tracks very closely to the ideal linear line: 2.00× at 2 cores, ~3.95× at 4 cores, ~7.87× at 8 cores. Parallel efficiency exceeds 98% up to 8 cores — the cluster hardware scales extremely well within node boundaries.

2. Parallelization limit found at 8 cores (2 nodes)

This directly answers Prof. Baun's challenge: "Find the point where the program cannot be parallelized anymore." On our cluster, the effective limit is 8 cores. Beyond this, MPI communication overhead at the 2→4 node boundary exceeds the compute gain. This is Amdahl's Law made visible — the serial fraction is the inter-node MPI_Reduce communication.

3. Problem size has marginal effect on the limit

We tested problem sizes from 1B to 25B (25× increase) to see if larger problems move the parallelization limit. Improvement at 8→16 cores grew from 0.4% to only 2.5% — the limit remains at 8 cores regardless. This confirms the limit is structural (network topology) not computational.

4. Amdahl vs Gustafson — problem size is key

At 1M points: speedup degrades rapidly beyond 4 cores (communication dominates — Amdahl's Law clearly visible). At 10B points: near-linear scaling to 8 cores, then the topology limit. This confirms Prof. Baun's lecture: "A sufficiently large problem can be parallelized efficiently."

5. Exceptional measurement consistency

Coefficient of variation < 0.5% for all 1B and 10B configurations across 3–5 runs. This demonstrates the cluster operates stably under controlled conditions — idle system, fan cooling, zombie process prevention. The consistency validates the results are real hardware behavior, not measurement noise.

Experiment Overview

Dense matrix multiplication C = A × B using MPI. Each rank receives rows of A via MPI_Scatter and a full copy of matrix B via MPI_Bcast. Results gathered with MPI_Gather. Communication complexity is O(N²) — as N grows, the broadcast of matrix B becomes larger, but O(N³) compute grows even faster, eventually allowing better scaling.

Tested across 8 matrix sizes (N=1000 to N=4000) and 6 core counts (1–32), 3–5 runs each. The investigation systematically found the memory limit and sweet spot.

Gustafson's Law — 32-Core Speedup vs Problem Size N

As N increases, O(N³) compute grows faster than O(N²) communication overhead — larger problems scale better. The crossover point is around N=3250 where the parallelization limit shifts past 8 cores. Green bar = sweet spot (N=3500).

Key Findings

⭐ Sweet spot: N=3500, 6.44× at 32 cores — best time-to-result

N=3500 completes in 68s at 32 cores with 6.44× speedup. While N=4000 achieves slightly better scaling (6.60×), it takes 100s at 32 cores. N=3500 gives the best balance of scaling efficiency and practical runtime. CV <1% confirms stable measurements.

✅ N=4000 scales better at 32 cores — MPI distributes memory pressure

N=4000 achieves 6.60× at 32 cores (best of all tested sizes). At 32 cores across 8 nodes, MPI_Scatter distributes rows of A across processes — each Pi3 handles less data. Matrix B (128MB) is broadcast to all processes but distributed compute means each Pi3 only runs 4 processes simultaneously, keeping memory manageable. At low core counts (1–4 cores on 1 node), all compute pressure falls on one Pi3 → slower scaling.

⚠️ Gustafson's Law clearly visible — scaling improves with N

32-core speedup grows consistently with N: 2.34× (N=1000) → 3.59× (N=1500) → 4.42× (N=2000) → 5.82× (N=3000) → 6.44× (N=3500) → 6.60× (N=4000). As N grows, O(N³) compute dominates O(N²) communication — larger problems parallelize more efficiently. This is Gustafson's Law: scale the problem with the cores.

❌ Memory limit at N=5000 — hard boundary confirmed

N=5000 causes immediate memory swap on Pi3: each process holds full matrix B (5000² × 8 = 200MB) × 4 processes per Pi3 = 800MB — exceeding Pi3's 1GB RAM. N=4000 (128MB × 4 = 512MB) runs successfully. Maximum safe N on this cluster: 4000. N=5000 is the confirmed hard limit.

🔍 Parallelization limit shifts with N (Amdahl vs Gustafson)

At N=1000: limit hits at 4 cores (efficiency collapses from 100% → 7.3% at 32 cores). At N=3250: crossover point — limit shifts past 8 cores. At N=4000: scaling continues improving to 32 cores. This confirms: fixed small problem = Amdahl dominates. Scaled large problem = Gustafson applies.

Experiment Overview

HPL (High Performance LINPACK) is the benchmark used to rank the TOP500 supercomputers worldwide. It solves a dense linear system Ax = b using LU factorization with partial pivoting, measuring sustained GFLOPS. The result must be numerically correct — it cannot be faked.

How it differs from Monte Carlo Pi: Monte Carlo has only ONE MPI_Reduce at the very end — communication is O(1), negligible. HPL requires a panel broadcast and pivot search across ALL nodes at every elimination step — communication is O(N²). This is why Monte Carlo plateaus at 8→16 cores while HPL drops at 4→8 cores. Same network boundary, amplified by a communication-heavy algorithm.

Monte Carlo Pi — O(1) communication

  • Each rank works completely independently
  • Single MPI_Reduce at the very end
  • Network barely involved during computation
  • Result: near-ideal scaling → plateau at 8→16 cores

HPL — O(N²) communication

  • Panel factorization: global pivot search every step
  • Panel broadcast to ALL nodes before continuing
  • Network heavily involved at every iteration
  • Result: GFLOPS drops 5.08→4.21 at 4→8 cores

Test Configuration

  • Tool: HPCC 1.5.0 (HPL component) via OpenMPI
  • Block size NB=128 (optimized for ARM Cortex-A53)
  • Pi3-only — Pi5 excluded (runs production stack)
  • Cooldown: 90s between tests · 3–5 runs per config

Problem Size N (scaled with cores)

  • 1c: N=5000 · 2c: N=5500 · 4c: N=6000
  • 8c: N=8000 · 16c: N=10000
  • 24c: N=13000 · 32c: N=15000
  • N scales to give more compute per message (Gustafson)

🔍 Why N scales with core count?

Each HPL process needs N²/P memory. More cores = more total RAM across nodes = larger N possible. Larger N means O(N³) compute grows faster than O(N²) communication — scaling improves. Same Gustafson principle as Matrix Multiply.

The 8-Core Dip — Prof. Baun's Key Question Answered

Prof. Baun's challenge: "Find the point where the program cannot be parallelized anymore." HPL shows something more revealing — it doesn't just plateau, it regresses and becomes unreliable.

⚠️ 8 cores: GFLOPS drops 5.082 → 4.212 AND 3/5 runs failed

At 4 cores (1 node): all MPI broadcasts stay in shared memory — zero latency. At 8 cores (2 nodes): every panel broadcast crosses the 100Mbit/s LAN switch. Out of 5 runs at 8 cores, only 2 passed HPL's residual verification check — the other 3 reported ERROR (numerical result incorrect due to communication errors). This is not just a performance dip — it's a reliability boundary.

Monte Carlo comparison — same boundary, different impact

Monte Carlo at 8→16 cores only plateaued (7.87×→7.99×) because it communicates once. HPL at 4→8 cores regresses AND fails because it communicates at every step. O(1) vs O(N²) communication makes all the difference.

Recovery at 16+ cores — Gustafson's Law

At 16 cores, N increases to 10000 — O(N³) compute grows faster than O(N²) communication. GFLOPS recovers to 6.28 and continues to 11.56 at 32 cores. All runs above 8 cores passed residual checks cleanly.

⚠️ N=6000 at 2 cores — memory crash

With only 2 cores, both processes run on the same Pi3. N=6000 required too much RAM per process, exceeding the ~600MB available. Reduced to N=5500 which succeeded. This shows the memory constraint is per-node, not per-core — fewer cores on one node means more memory per process.

❌ N=8000 at 8 cores — communication boundary

Memory fit (~512MB across 2 nodes). But 8 cores is the first inter-node boundary — sustained HPL load caused MPI communication errors under the 100Mbit/s network. 3 out of 5 runs failed HPL's residual check. Not a memory issue — a network reliability issue under heavy sustained load.

Bottleneck Analysis — What Limited Our Results?

Our cluster faced three distinct bottlenecks during HPL benchmarking. Understanding them explains both the dip and the overall efficiency ceiling.

❌ Bottleneck 1: 100Mbit/s LAN at first node boundary (8 cores)

HPL's O(N²) panel broadcasts crossed the 100Mbit/s switch at every elimination step. The network simply couldn't sustain the required bandwidth and latency under full load — causing the GFLOPS dip and 3/5 residual failures at 8 cores. Fix: 1Gbit/s or 10Gbit/s network would eliminate this bottleneck entirely. A single switch connecting all nodes (vs our 2-switch setup) would also reduce latency.

⚠️ Bottleneck 2: Pi3 memory (~600MB available) limited problem size N

With only ~600MB free per Pi3 (after OS, k3s stopped), we couldn't run larger N values. Larger N is essential for HPL — it gives O(N³) compute more room to dominate O(N²) communication. N=6000 at 2 cores and N=11000 at 16 cores both crashed due to this limit. Fix: Pi4 (4GB RAM) or Pi5 (8GB RAM) nodes would allow N=20000–50000, dramatically improving scaling efficiency and pushing GFLOPS much higher.

⚠️ Bottleneck 3: ARM Cortex-A53 software floating point

Pi3's CPU has no hardware double-precision FPU — HPL's double-precision matrix operations are computed in software, severely limiting raw GFLOPS per core. This caps our theoretical efficiency regardless of network or memory improvements. Fix: ARM Cortex-A72 (Pi4) or Cortex-A76 (Pi5) have hardware FPU — estimated 3–5× improvement in raw GFLOPS per core.

🔍 What could we achieve with better hardware?

With 8× Raspberry Pi 5 (8GB RAM, Cortex-A76, Gigabit LAN): N could scale to ~40000–50000 at 32 cores, network latency would drop 10×, and FPU hardware would deliver 3–5× more GFLOPS per core. Estimated peak: 50–80 GFLOPS vs our current 11.56 GFLOPS — a 4–7× improvement purely from hardware upgrade, same software stack.

Key Findings

✅ Peak: 11.56 GFLOPS at 32 cores — strong scaling above 8 cores

After the dip: 4.21 (8c) → 6.28 (16c) → 8.58 (24c) → 11.56 (32c). The 16→32 jump achieves 1.84×. Excluding the 8-core anomaly, HPL scales well across all boundaries on 100Mbit/s LAN.

⚠️ HPL efficiency: 25.8% of theoretical peak

Theoretical peak for 32× ARM Cortex-A53 @ 1.4GHz ≈ 44.8 GFLOPS. Achieved: 11.56 GFLOPS = 25.8% efficiency. Typical for ARM SBC clusters without hardware FPU. Prof. Baun's own published research on similar clusters reports comparable efficiency figures.

❌ 8-core boundary: dip + 3/5 residual failures

Only 2 out of 5 runs at 8 cores passed HPL's residual check — confirming the first inter-node boundary is a real reliability issue on 100Mbit/s LAN under sustained HPL load, not measurement noise.

📊 Pi3-02 power instability under sustained HPL load

Pi3-02 spontaneously reboots at N≥8000 — confirmed power supply limitation. Mitigated by placing it last in MPI node order so it only participates at 32 cores.
